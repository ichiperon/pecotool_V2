# PecoTool V2 追加機能要件書

## 概要

本ドキュメントは、PecoTool V2に追加する2つの機能の要件を定義する。

| # | 機能名 | 難易度 |
|---|--------|--------|
| 1 | 一括序列修正機能 | ★★☆ |
| 2 | PDF保存時の圧縮機能 + 推定容量表示 | ★★☆ |

---

## 機能1: 一括序列修正機能

### 目的

現在、BBの並び順（序列）を変更するにはOcrEditorでカードを1つずつドラッグ&ドロップする必要がある。  
キャンバス上でAlt+ドラッグにより方向を指定し、その方向に基づいて全BBの`order`を一括で振り直すことで、序列修正の効率を大幅に向上させる。

### 操作フロー

1. ユーザーがキャンバス上で **Alt+ドラッグ** を開始する
2. ドラッグ中、始点から現在のマウス位置まで **矢印が描画** される（視覚フィードバック）
3. 矢印の近くに **ソート方向のラベル** を表示する（例: 「↓ 上→下」）
4. ドラッグを離すと、矢印の方向に基づいて現在ページの全BBの`order`が再計算される
5. 右カラム（OcrEditor）のカード並び順が即座に更新される

### 要件一覧

#### REQ-1.1: 矢印描画（視覚フィードバック）

- Alt+ドラッグ中、キャンバスのオーバーレイ上に始点→終点の矢印を描画する
- 矢印は方向がわかるデザインとする（先端に三角形等）
- 矢印の近くに現在のソート方向をテキストで表示する（例: 「↓ 上→下」「← 右→左」等）
- ドラッグ終了後、矢印・ラベルは消去する
- 描画モード・分割モード中はAlt+ドラッグによる一括序列修正を無効とする（既存操作との競合回避）

#### REQ-1.2: 方向判定

- ドラッグの始点・終点から方向ベクトルを算出する
- 方向は以下の8方向に分類する（45度刻み）:
  - 上、下、左、右
  - 右上、右下、左上、左下（斜め方向）

#### REQ-1.3: ソートルール

方向に応じて、現在表示中ページの**全BB**を対象に`order`を0からの連番で振り直す。  
**ソートの基準座標は各BBの中心座標（`x + width/2`, `y + height/2`）を使用する。**  
ソート実行後、ページの`isDirty`を`true`にする（PDF保存時のメタデータに`order`が含まれるため）。

| 方向 | 主軸ソート | 副軸ソート |
|------|-----------|-----------|
| 上→下 | Y座標 昇順 | X座標 昇順 |
| 下→上 | Y座標 降順 | X座標 降順 |
| 左→右 | X座標 昇順 | Y座標 昇順 |
| 右→左 | X座標 降順 | Y座標 降順 |
| 左上→右下 | Y座標 昇順 | X座標 昇順 |
| 右下→左上 | Y座標 降順 | X座標 降順 |
| 右上→左下 | Y座標 昇順 | X座標 降順 |
| 左下→右上 | Y座標 降順 | X座標 昇順 |

#### REQ-1.4: 同一行/同一列の判定（グループ化閾値）

- 主軸方向でBBをソートする際、主軸の中心座標が近いBBは**同一行（または同一列）**として扱い、副軸でソートする
- 「近い」の判定基準: **現在ページの全BBの主軸方向サイズ（heightまたはwidth）の全体平均値**に対する割合で判定する
  - 例: ページ全体のBBの平均高さが60px、閾値50%なら、Y中心座標の差が30px以内のとき同じ行とみなす
  - ※比較する2BBペアごとの個別平均ではなく、ページ全体の平均値を一律で使用する
- **デフォルト値: 50%**
- この閾値はツール上で変更可能とする（REQ-1.5参照）

#### REQ-1.5: 閾値設定UI

- ツールバーまたは設定エリアに、同一行/列判定の閾値を変更するUIを設置する
- 入力方式: スライダーまたは数値入力（0%〜100%）
- デフォルト値: 50%
- 変更は次回のAlt+ドラッグ操作から適用される（閾値変更だけでは並び替えは発生しない）
- 設定値はアプリ再起動後も保持する（localStorage等）

#### REQ-1.6: ショートカット

- **Alt+ドラッグ** が本機能の操作ショートカットである
- 専用のキーボードショートカットやツールバーボタンは設けない（Alt+ドラッグのみで操作完結）
- ショートカットヘルプ（右クリックメニュー）に `Alt+ドラッグ → 一括序列` を追加する

#### REQ-1.7: Undo/Redo対応

- 一括序列変更はUndo/Redo操作の対象とする
- 既存のページスナップショット方式（`undoStack` / `redoStack`）で対応可能

#### REQ-1.8: エッジケース

- BBが0件の場合: 何もしない
- BBが1件の場合: `order = 0`を設定して終了
- ドラッグ距離が極端に短い場合（5px未満等）: 操作を無視する（誤操作防止）

## 機能2: PDF保存時の圧縮機能 + 推定容量表示

### 目的

現在、PDF保存は非圧縮固定のため、ファイルサイズが大きくなる傾向がある。  
保存時に圧縮の有無を選択可能にし、保存前に推定ファイルサイズを確認できるようにすることで、ユーザーがファイルサイズを意識した保存が行える。

### 操作フロー

**通常保存（Ctrl+S）:**
1. 前回選択した圧縮オプションで即座に保存する（ダイアログなし）
2. 保存完了後、トーストにファイルサイズを表示する（例: 「保存しました。(3.2 MB)」）

**名前を付けて保存（Ctrl+Shift+S）:**
1. 圧縮オプション選択ダイアログを表示する
2. ダイアログ内で非圧縮/圧縮を選択し、推定サイズを確認する
3. 「保存」を押すと Tauriのファイルパス選択ダイアログを表示する
4. ファイルパスを確定後、選択したオプションで保存する
5. 保存完了後、トーストにファイルサイズを表示する

### 要件一覧

#### REQ-2.1: 圧縮オプション

- 以下の3つの選択肢を提供する:
  - **非圧縮**: 現在と同じ動作（`useObjectStreams: false`）
  - **圧縮 (Object Streams)**: pdf-libのオブジェクトストリーム圧縮を有効化（`useObjectStreams: true`）
  - **高圧縮 (ラスタライズ)**: 背景PDFを全ページJPEG画像化して再構築。ファイルサイズを大幅削減できるが処理が重く画質が低下する。JPEG画質はスライダーで調整可能（10〜100%、デフォルト60%）
- デフォルト選択: **非圧縮**（既存動作との互換性維持）

> **補足**: `useObjectStreams`はオブジェクトメタデータの圧縮が主であり、画像データ自体の再圧縮は含まない。画像が主体のPDFでは圧縮効果が限定的（数%程度）になる場合がある。高圧縮（ラスタライズ）は効果が大きいが不可逆処理であり、使用には注意を要する。この制約をダイアログ内に注記として表示する。

#### REQ-2.2: 圧縮オプション選択ダイアログ（名前を付けて保存のみ）

- **Ctrl+Shift+S（名前を付けて保存）時のみ**ダイアログを表示する
- **Ctrl+S（通常保存）はダイアログを表示せず、前回の選択で即保存する**
- ダイアログの構成要素:
  - 圧縮オプションの選択UI（ラジオボタン: 非圧縮/圧縮/高圧縮の3択）
  - 高圧縮選択時はJPEG画質スライダー（10〜100%）を追加表示する
  - 推定ファイルサイズの表示エリア（算出中はスピナーを表示）
  - 高圧縮オプションの推定サイズは「推定不可 (実行時に計算)」と表示する
  - 圧縮効果の制約に関する注記（例: 「※標準以下の圧縮では画像データは再圧縮されません」）
  - 「保存する」ボタン（算出中・算出完了どちらでも常に有効）
  - 「キャンセル」ボタン
- ダイアログはモーダル表示とする

#### REQ-2.3: 推定ファイルサイズの表示

- ダイアログ表示と同時に非同期でサイズ算出を開始する
- 算出方法:
  1. `PDFDocument`の構築（フォント埋め込み・ページ描画）は**1回だけ**行う
  2. 構築済みの`PDFDocument`に対して`save()`を非圧縮・圧縮の2通りのオプションで呼び出し、それぞれのバイト列を取得する
  3. 各`Uint8Array.byteLength`を表示する
- 表示フォーマット: 人間が読みやすい形式（例: `3.2 MB`、`850 KB`）
- 両方のサイズを並べて表示し、削減率も併記する
  - 例: `非圧縮: 5.2 MB / 圧縮: 3.1 MB（40%削減）`
- 算出に失敗した場合は「サイズ推定不可」と表示する

#### REQ-2.4: サイズ算出中のUX

- 算出中はサイズ表示エリアに「計算中...」とローディング表示を行う
- 「保存」ボタンは算出中・算出完了のいずれでも常に有効とする
  - 算出完了を待たずに保存を実行できる
- 算出に失敗した場合は「サイズ推定不可」と表示し、保存ボタンは有効のままとする

#### REQ-2.5: `pdfSaver.ts`のインターフェース変更

- 圧縮オプションを引数で受け取れるようにシグネチャを変更する:
  ```ts
  async function savePDF(
    originalPdfBytes: Uint8Array,
    documentState: PecoDocument,
    compression: 'none' | 'compressed' | 'rasterized',
    rasterizeQuality: number  // 0.0〜1.0、rasterized時のJPEG画質
  ): Promise<Uint8Array>
  ```
- `compression`未指定時は`'none'`（`useObjectStreams: false`）を維持する
- `rasterized`時は`buildRasterizedPdfDocument()`を呼び出し、全ページをJPEG画像化して再構築する
- サイズ推定用に`buildPdfDocument()`（PDFDocument構築のみ）を内部関数として分離し、`estimateSizes()`から`save()`を2通りのオプションで呼び出す

#### REQ-2.6: 保存後のサイズ表示

- 保存完了後の既存トーストメッセージにファイルサイズと圧縮方式を付記する
  - 通常保存: 「保存しました。(3.2 MB)」
  - 名前を付けて保存（非圧縮）: 「名前を付けて保存しました。(3.2 MB・非圧縮)」
  - 名前を付けて保存（圧縮）: 「名前を付けて保存しました。(2.1 MB・標準圧縮)」
  - 名前を付けて保存（ラスタライズ）: 「名前を付けて保存しました。(1.0 MB・高圧縮)」
- 実際に書き込んだバイト列のサイズを使用する
- ラスタライズ時は処理が重いため、保存開始時にトーストで「高圧縮処理中です(画質XX%)...しばらくお待ち下さい」を表示する

#### REQ-2.7: 設定の永続化

- 前回選択した圧縮オプションを`localStorage`に保存する
- アプリ再起動後も保持し、次回の通常保存・ダイアログのデフォルト選択に反映する
- キー名: `peco-save-compression`、値: `"none"` / `"compressed"` / `"rasterized"`
- ラスタライズ画質も別途保存する
  - キー名: `peco-rasterize-quality`、値: `"10"`〜`"100"`（整数の文字列）、デフォルト `"60"`

## 技術メモ

### 機能1に関する注意点

- `PdfCanvas.tsx`のイベントハンドラにAlt+ドラッグの処理を追加する
- 既存のドラッグ操作との優先順位:
  - **Alt+ドラッグ**: 一括序列修正（新規）— Alt+ホイール（ズーム）との競合なし（ドラッグ vs ホイール）
  - **Ctrl+クリック/ドラッグ**: 複数選択（既存）— 競合解消済み
  - **Space+ドラッグ**: パン（既存）
  - **描画モード+ドラッグ**: BB新規作成（既存）
- ソート後の`order`更新は`pecoStore.ts`の`updatePageData`経由で行う

### 機能2に関する注意点

- pdf-libの`useObjectStreams: true`はPDF 1.5以降の仕様。古いPDFリーダーでの互換性に注意
- サイズ推定は`PDFDocument`構築を1回に抑え、`save()`呼び出しのみ2回行うことでパフォーマンスを確保する
- `save()`を2回呼び出した後、片方のバイト列はサイズ取得後に参照を解放してGCに委ねる
- 実際の保存処理では推定時に生成したバイト列を再利用できれば理想だが、選択肢が変わる可能性があるため選択確定後に再度`save()`を1回呼ぶ方が安全
- 現在の`handleSave` / `handleSaveAs`（App.tsx）はどちらも`savePDF()`を直接呼ぶ構造。REQ-2.5のシグネチャ変更後は引数を追加するだけで対応可能
