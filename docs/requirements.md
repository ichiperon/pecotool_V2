# PecoTool V2 — 要件定義書

## 1. プロジェクト概要

### 1.1 目的
PDF に付与された OCR テキスト（透明テキストレイヤー）を、メモ帳で文字を直すような感覚で自由に編集できるデスクトップツールを開発する。

### 1.2 背景・課題
- 既存の OCR 編集ツールは「再 OCR」「文字読み取り」が中心であり、手動で誤字脱字を直す機能に特化したツールが存在しない。
- スキャン PDF の OCR 品質が低い場合、1 文字ずつバラバラのバウンディングボックスになる・読み順が狂うなどの問題があるが、これを手軽に修正する手段がない。
- 数千ページ規模の PDF を扱う業務で、既存ツールではメモリ逼迫やフリーズが発生する。

### 1.3 対象ユーザー
- 一般家庭の PC を使用するユーザー（高スペックマシンを前提としない）。
- PDF の OCR 品質を手動で改善したいユーザー（文書管理者、アーカイブ担当者等）。
- 技術的な専門知識を持たない一般ユーザーでも操作できることを前提とする。

### 1.4 前提・制約

| 項目 | 内容 |
|------|------|
| 編集の定義 | 再 OCR や文字認識機能ではなく、既存 OCR テキストの手動修正（誤字脱字の手打ち修正） |
| パフォーマンス要件 | 数千ページ規模の PDF でもフリーズ・遅延なく動作すること |
| メモリ制約 | 一般家庭 PC（RAM 8–16 GB 想定）で問題なく動作すること |
| 対象 OS | Windows（最低限）。可能であればクロスプラットフォーム |
| 対象 PDF | OCR テキストレイヤー付きの PDF（画像 + 透明テキスト構成） |
| 対象外 | テキストベース PDF（Word 等から生成された PDF）の編集は対象外 |
| 同時オープン | 1 ファイルずつの編集を想定（マルチドキュメントは対象外） |

### 1.5 スコープ外
- OCR エンジンの搭載（文字認識処理そのものは行わない）
- PDF のページ追加・削除・回転等のページ操作
- PDF の画像レイヤー編集（墨消し・注釈等）
- PDF のパスワード保護・暗号化処理
- クラウド連携・マルチユーザー機能

#### F-01: PDF 読み込み・表示

**概要**
数百〜数千ページの OCR 付き PDF を読み込み、ページ画像を表示する。

**詳細仕様**

| 項目 | 仕様 |
|------|------|
| 対応形式 | PDF 1.3〜2.0。OCR テキストレイヤー（透明テキスト）を含むもの。※ PDF 1.3 は業務用スキャナーで生成される場合があるため対象に含める |
| ファイルオープン方法 | ① メニューバーの「ファイル → 開く」または Ctrl+O ② PDF 画像表示エリアへのドラッグ＆ドロップ ③ ファイルの関連付け（ダブルクリック） |
| 最大ページ数 | 制限なし（メモリ管理により実質的に数千ページ対応） |
| 最大ファイルサイズ | 制限なし（遅延読み込みによりファイルサイズに依存しない） |

**読み込み戦略**

```
1. ファイルオープン時
   ├─ PDF メタデータ（総ページ数・ページサイズ等）を即座に取得
   ├─ 1 ページ目の画像 + OCR データを読み込み → 表示
   └─ サムネイルは可視範囲のみ生成開始（バックグラウンド）

2. ページ遷移時
   ├─ 表示対象ページの画像 + OCR データを取得
   ├─ 前後 ±1 ページをプリフェッチ（バックグラウンド）
   └─ 表示範囲外のページデータを LRU ポリシーで解放

3. サムネイル
   ├─ 仮想スクロール：可視範囲 + 上下バッファ分のみ DOM に描画
   └─ サムネイル画像は低解像度（長辺 150px 程度）で生成しキャッシュ
```

**受入条件**
- [ ] 1,000 ページの PDF を開いて 3 秒以内に 1 ページ目が表示される
- [ ] 1,000 ページ表示時のメモリ使用量が 500 MB 以下
- [ ] ページ遷移が 200 ms 以内に完了する
- [ ] ドラッグ＆ドロップで PDF を開ける
- [ ] OCR テキストレイヤーのない PDF を開いた場合、画像のみ表示しエラーにならない

**エラーハンドリング**
- 破損 PDF：「ファイルを読み込めません。PDF が破損している可能性があります。」と表示
- パスワード付き PDF：「パスワード付き PDF には対応していません。」と表示
- 非 PDF ファイル：「PDF ファイルを選択してください。」と表示

#### F-02: OCR テキスト編集

**概要**
PDF の透明テキストレイヤーから OCR テキストを抽出し、テキストエリアに表示。手動で修正できる。

**詳細仕様**

| 項目 | 仕様 |
|------|------|
| 表示単位 | バウンディングボックス単位で区切って表示 |
| 編集操作 | テキストのインライン編集（クリックでフォーカス → 手打ち修正） |
| 対応文字方向 | 縦書き（vertical-rl）・横書き（horizontal-tb） |
| 文字方向の判定 | バウンディングボックスの縦横比 + テキストの配置方向から自動判定 |
| 文字方向の表示 | テキスト編集エリアでは横書き表示に統一（編集のしやすさ優先）。ただし縦書き/横書きのラベルを表示 |
| 編集中の視覚フィードバック | 編集中のバウンディングボックスを PDF 表示エリアでハイライト |
| 未保存変更の表示 | 変更があるバウンディングボックスに「●（未保存マーク）」を表示 |

**ユーザー操作フロー**

```
1. ページを表示すると、右エリアに OCR テキストがバウンディングボックス単位で一覧表示される
2. 修正したいテキストブロックをクリック → 編集モードに入る
3. テキストを手打ちで修正する（メモ帳と同じ感覚）
4. Enter または別のブロックをクリックで確定
5. 確定された変更は Undo/Redo スタックに積まれる
6. 保存操作で PDF に反映
```

**受入条件**
- [ ] 各ページの OCR テキストがバウンディングボックス単位で表示される
- [ ] テキストをクリックして手打ちで修正できる
- [ ] 縦書き OCR のテキストも正しく表示・編集できる
- [ ] 編集中のテキストに対応する PDF 上のバウンディングボックスがハイライトされる
- [ ] 空のバウンディングボックス（テキストなし）も他の BB と同様にカード形式で表示される（プレースホルダー文字や折りたたみ表示は行わない）

#### F-03: バウンディングボックスのグループ化

**概要**
1 文字ずつ分かれたバウンディングボックスを選択し、1 つのバウンディングボックスに統合する。

**詳細仕様**

| 項目 | 仕様 |
|------|------|
| 選択方法 | ① テキスト編集エリアで Shift+クリック / Ctrl+クリックで複数選択 ② PDF 表示エリアでドラッグ矩形選択 |
| グループ化実行 | 選択後に右クリックメニューまたはツールバーの「グループ化」ボタン |
| 座標の統合 | 選択されたバウンディングボックスの外接矩形を新しいバウンディングボックスとする |
| テキストの統合順序 | **選択した BB を現在のテキスト編集エリアの表示順（=読み順 `order`）で結合する**。選択方法（Ctrl+クリック / Shift+クリック / ドラッグ矩形）による順序の差異は無く、常に表示順で一意に決まる |
| 文字方向 | 統合対象の先頭ブロックの方向を継承（混在時も警告は行わず、先頭の方向に揃える） |
| 統合上限 | 制限なし |
| グループ化後の BB 座標調整 | グループ化後も PDF 画像上でドラッグして位置・サイズを手動調整可能（F-07 の位置調整と同じ操作） |

> 結合順序を変えたい場合は、グループ化前に F-06 の並び順変更で対象ブロックを希望の順序に並べ替えてから実行する。

**グループ化の例**

```
Before:
  [BB-1] "あ"  (x:10, y:10, w:20, h:20)
  [BB-2] "い"  (x:30, y:10, w:20, h:20)
  [BB-3] "う"  (x:50, y:10, w:20, h:20)

After (BB-1, BB-2, BB-3 を選択してグループ化):
  [BB-1'] "あいう"  (x:10, y:10, w:60, h:20)  ← 外接矩形
```

**受入条件**
- [ ] 複数のバウンディングボックスを選択してグループ化できる
- [ ] グループ化後のテキストは上記ルールに従った順序で結合される
- [ ] グループ化後の座標は外接矩形になる
- [ ] グループ化後に BB の位置・サイズを手動調整できる
- [ ] グループ化は Undo で元に戻せる（= 分割操作として機能する）
- [ ] 1 つのバウンディングボックスのみ選択時は「グループ化」が無効（グレーアウト）

#### F-04: OCR テキスト削除

**概要**
指定したページまたは範囲の OCR テキストを一括削除する。

**詳細仕様**

| 項目 | 仕様 |
|------|------|
| 削除単位 | ① 個別バウンディングボックス ② 現在のページ全体 ③ 全ページ |
| 個別削除 | テキスト編集エリアで選択 → Delete キーまたは右クリック「削除」 |
| ページ単位削除 | ツールバーの「OCR消去」ドロップダウンから「現在のページ」を選択 |
| 全ページ削除 | ツールバーの「OCR消去」ドロップダウンから「全ページ」を選択 |
| 確認 | ページ単位以上の削除は確認ダイアログを表示。全ページ削除は「この操作は元に戻せません」の警告を追記 |
| 結果 | 削除後は OCR なしの画像のみのページとなる |

**受入条件**
- [ ] 個別のバウンディングボックスを削除できる
- [ ] ページ全体の OCR を一括削除できる
- [ ] 全ページの OCR を一括削除できる
- [ ] 削除操作は Undo で元に戻せる
- [ ] ページ単位以上の削除時に確認ダイアログが表示される

#### F-05: テキスト並び順の整合性保証

**概要**
ツール内で表示されるテキストの並び順と、保存後の PDF からコピペした際のテキスト順序を完全に一致させる。

**詳細仕様**

| 項目 | 仕様 |
|------|------|
| 順序の定義 | テキスト編集エリアでの上から下への並び順 = PDF テキストレイヤーの読み順 |
| 順序の決定方法 | **Content Stream 内のテキストオブジェクト出現順を正とする**。Structure Tree が存在する場合でも Content Stream 順を優先する（理由: コピペ順序を直接制御するのは Content Stream であり、Structure Tree は Acrobat 等のアクセシビリティ向け情報のため） |
| 検証方法 | 保存後の PDF を Acrobat 等で開き Ctrl+A → Ctrl+C した結果がツール内の順序と一致 |
| プレビュー機能 | テキスト編集エリアの表示がそのままコピペ結果のプレビューとなる |

**コピペ順序の制御**

```
PDF テキストレイヤーの構造:

  /Content Stream
    ├─ TextBlock[0] "タイトル"       ← コピペ時 1 番目
    ├─ TextBlock[1] "本文1行目"      ← コピペ時 2 番目
    ├─ TextBlock[2] "本文2行目"      ← コピペ時 3 番目
    └─ TextBlock[3] "ページ番号"     ← コピペ時 4 番目

  ↕ この順序がテキスト編集エリアの並び順と一致する
```

**受入条件**
- [ ] テキスト編集エリアの並び順で保存後の PDF がコピペされる
- [ ] 読み込み直後の並び順は元の PDF のコピペ順と一致する
- [ ] 並べ替え後に保存 → 再度開いた際に並べ替え後の順序が保持されている

#### F-06: バウンディングボックスの並び順変更

**概要**
バウンディングボックスの読み順を変更する。個別移動（ドラッグ＆ドロップ）と一括序列修正（Alt+ドラッグ）の 2 つの操作方法がある。

**詳細仕様**

| 項目 | 仕様 |
|------|------|
| 操作方法①（個別） | テキスト編集エリアでバウンディングボックスをドラッグ＆ドロップ |
| 複数選択移動 | Shift/Ctrl 選択した複数ブロックを一括で移動可能 |
| 視覚フィードバック（個別） | ドラッグ中はドロップ先を示すインジケーター（挿入線）を表示 |
| 即時反映 | 並べ替えは即座にプレビューに反映（保存は別操作） |
| 操作方法②（一括序列修正） | PDF 表示エリアで **Alt+ドラッグ** することで、ドラッグ方向に基づいて現在ページの全 BB の `order` を一括再計算 |
| 方向判定 | 360度を8方向（上下左右・斜め4方向）に分類 |
| ソートルール | 各 BB の中心座標を基準に、主軸・副軸の2段階ソートで並べ替え |
| 同一行/列判定 | ページ内の全 BB の平均サイズに基づく「判定閾値（デフォルト 20%）」を使用。閾値内の中心座標差は同一行/列とみなし副軸ソートを適用。閾値は設定ドロップダウンから変更可能（localStorage 保存） |
| 視覚フィードバック（一括） | ドラッグ中に始点から終点へのオレンジ色の点線矢印と方向ラベル（例: 「↓ 上→下」）をリアルタイム表示 |

**ユースケース**

```
Before: "画像キャプション" → "本文" → "タイトル" → "脚注"
（OCR が読み取り順を誤っている）

操作: "タイトル" を先頭にドラッグ、"脚注" を末尾にドラッグ

After: "タイトル" → "本文" → "画像キャプション" → "脚注"
（正しい読み順に修正）
```

**受入条件**
- [ ] ドラッグ＆ドロップでバウンディングボックスの順序を変更できる
- [ ] 複数ブロックを一括で移動できる
- [ ] Alt+ドラッグで現在ページの全 BB を一括で序列修正できる
- [ ] Alt+ドラッグ中にオレンジ色の矢印と方向ラベルが表示される
- [ ] 並べ替えは Undo で元に戻せる
- [ ] 並べ替え後に保存した PDF のコピペ順が新しい順序を反映している

#### F-07: バウンディングボックスの追加・手打ち入力

**概要**
PDF 画像上の任意の位置に新しいバウンディングボックスを追加し、テキストを入力する。

**詳細仕様**

| 項目 | 仕様 |
|------|------|
| 追加方法 | ツールバーの「追加」ボタン（または Ctrl+F10）で追加モードへ切替え、PDF 画像上でドラッグして矩形を作成 |
| テキスト入力 | 矩形作成後、テキスト編集エリアに新規ブロックが追加されフォーカスされる |
| 位置調整 | 追加後も PDF 画像上でドラッグして位置・サイズを調整可能。**既存 BB（読み込み時から存在するもの）も同様に位置・サイズ調整が可能** |
| 読み順 | 新規ブロックは BB の座標位置から読み順を自動推定して挿入する（横書き: 左上→右下、縦書き: 右上→左下）。推定位置が不適切な場合はドラッグで任意の位置に移動可能（F-06）。自動推定が困難な場合（既存 BB と座標が大きく離れている等）は末尾に追加 |
| 文字方向 | 追加時に縦書き/横書きを選択（デフォルト: 横書き） |
| 既存 BB とのグループ化 | 追加した BB を既存の BB と選択してグループ化可能（F-03） |

**ユーザー操作フロー**

```
1. ツールバーの「追加」ボタンをクリック → カーソルが十字に変わる
2. PDF 画像上でドラッグして矩形を描画
3. テキスト編集エリアに空のブロックが追加される
4. テキストを手打ちで入力
5. 必要に応じて読み順をドラッグで調整
6. 必要に応じて既存 BB とグループ化
```

**受入条件**
- [ ] PDF 画像上でドラッグして新規バウンディングボックスを作成できる
- [ ] 作成後にテキストを入力できる
- [ ] 追加した BB の位置・サイズを後から調整できる
- [ ] 追加した BB を既存の BB とグループ化できる
- [ ] 追加操作は Undo で元に戻せる

#### F-08: PDF 保存

**概要**
編集内容を反映した PDF を保存する。保存後の PDF は編集後のテキストレイヤーで検索・コピペが可能。

**詳細仕様**

| 項目 | 仕様 |
|------|------|
| 保存方法 | ① 上書き保存（Ctrl+S） ② 名前を付けて保存（Ctrl+Shift+S） |
| 保存対象 | テキストレイヤー（バウンディングボックス座標 + テキスト + 読み順）のみ |
| 画像レイヤー | 一切変更しない（無劣化で保持） |
| 保存処理 | 一時ファイルに書き出し → 検証 → リネームで置き換え（書き込み中断による破損を防止） |
| 進捗表示 | 大容量 PDF の保存時はプログレスバーを表示 |
| 初回保存 | 新規に「名前を付けて保存」ダイアログを表示 |
| 保存と Undo の関係 | 保存後も Undo 履歴は保持する。保存後に Undo で保存前の状態に戻した場合、isDirty フラグが再び true になる（未保存マーク再表示）。その状態で Ctrl+S すると Undo 後の状態で上書き保存される。**保存操作自体は Undo 対象外**であり、「保存を取り消す」機能は提供しない。また、初期読み込みや表示用のページデータマージ（`isDirty: false` 指定時）では全体の `isDirty` フラグは更新されない。 |

**名前を付けて保存の保存形式**

現状の「名前を付けて保存（Ctrl+Shift+S）」は、OS 標準の保存先選択ダイアログのみを表示し、PDF は常に **非圧縮（`useObjectStreams: false`、旧形式 xref 維持）** で書き出される。pdf-lib の `save()` による全体書き出し後、`restorePdfVersion` で元 PDF のバージョンヘッダーを書き戻すことで、PDF バージョン・互換性を保持する（Acrobat 7.0 互換を維持）。

- **保存後フィードバック**: 完了時のトーストメッセージに実際のファイルサイズを表示する（`保存しました。(123.4 KB)` など）

> 補足: ソースツリーには SaveDialog コンポーネント（非圧縮 / Object Streams 圧縮 / 高圧縮ラスタライズの 3 択 UI と JPEG 画質スライダー）が含まれており、`rasterize.worker.ts` ではラスタライズ保存処理も実装済みだが、現時点では保存フロー（`useFileOperations.executeSaveAs`）から呼び出されていない（将来接続予定）。

**保存時の処理フロー**

```
1. 編集データの整合性チェック
   ├─ 空テキストの BB が存在する場合 → 警告（「空のテキストブロックがあります。保存しますか？」）
   └─ 座標が重複する BB → そのまま保存（警告なし）

2. 一時ファイルへの書き出し
   ├─ 元 PDF を複製
   ├─ 各ページのテキストレイヤーを編集後データで再構築
   │   ├─ テキスト内容の更新
   │   ├─ バウンディングボックス座標の更新
   │   └─ 読み順（Content Stream 内の順序）の更新
   └─ 画像レイヤーはストリームをそのままコピー（無変換）

3. 検証
   └─ 一時ファイルが正常な PDF として読み込めるか検証

4. リネーム
   └─ 一時ファイルを正式ファイル名にリネーム（アトミック操作）
```

**受入条件**
- [ ] Ctrl+S で上書き保存できる
- [ ] Ctrl+Shift+S で名前を付けて保存できる

| サムネイルスクロール | 60 fps | 1,000 ページ分のサムネイルを高速スクロール中 |
| テキスト編集レスポンス | 入力遅延なし | 100 個の BB を持つページでテキストを編集中 |
| 保存処理 | 30 秒以内 | 1,000 ページ PDF の保存完了 |
| アプリ起動 | 2 秒以内 | ファイル未オープン状態での起動完了 |

### 3.2 パフォーマンス実現方針

#### メモリ管理戦略

```
                    メモリ上に保持
              ┌──────────────────────┐
              │                      │
  ... [P3] [P4] | [P5] [★P6] [P7] | [P8] [P9] ...
              │   バッファ  表示中  バッファ │
              └──────────────────────┘
                 他は LRU キャッシュで管理
                 （キャッシュ上限: 50 ページ分）
```

- **アクティブウィンドウ**: 表示中ページ ± 2 ページ（計 5 ページ）を高解像度でメモリ保持
- **LRU キャッシュ**: 直近アクセスした 50 ページ分のレンダリング結果をキャッシュ
- **キャッシュ超過時**: 最も古いアクセスのページから画像データを解放
- **OCR データ**: 表示中ページの OCR データのみメモリ保持。編集済みページの差分は軽量な JSON で保持
- **サムネイル**: 低解像度（長辺 150px）で生成。全ページ分をキャッシュしても数十 MB 程度

#### スレッド戦略

| スレッド | 担当処理 |
|---------|---------|
| メインスレッド（UI） | ユーザー操作の受付・画面描画・テキスト編集 |
| Worker 1: PDF レンダラー | ページ画像のレンダリング（プリフェッチ含む） |
| Worker 2: サムネイル生成 | サムネイル画像のバックグラウンド生成 |
| Worker 3: PDF 保存 | 保存処理（メインスレッドをブロックしない） |

### 3.3 信頼性

| 項目 | 仕様 |
|------|------|
| 保存の原子性 | 一時ファイル → リネーム方式で保存中断による破損を防止 |

#### 自動バックアップ

| 項目 | 仕様 |
|------|------|
| 保存間隔 | デフォルト 5 分間隔のタイマーで起動。`requestIdleCallback` が利用可能な環境では UI アイドル時のみ実行（タイムアウト 30 秒） |
| 実行条件 | ドキュメントが未保存変更を持つ場合のみ実行。Dirty なページが 1 つも無ければスキップ |
| 保存先 | Tauri バックエンド（Rust 側）が管理するアプリ固有のバックアップ領域に書き出す |
| バックアップ内容 | Dirty な PageData（textBlocks 含む。サムネイルは除外）を JSON 化し、Rust の `save_backup` コマンドで永続化 |
| 削除タイミング | ① ユーザーが明示的に保存（Ctrl+S / 名前を付けて保存）した時点で `clear_backup` を呼んで削除 ② 「名前を付けて保存」時は旧パス・新パス両方のバックアップを削除 |
| クラッシュリカバリ | 起動時に `check_pending_backups` を呼び、未処理バックアップがあれば BackupRestoreDialog で「復旧する / 破棄する」を選択 |

### 3.4 ユーザビリティ

| 項目 | 仕様 |
|------|------|
| 初見操作 | マニュアルなしで基本操作（PDF 読み込み → テキスト編集 → 保存）が完了できる |
| フィードバック | 全ての操作に対して即座に視覚的フィードバックを返す |
| エラー回復 | 全ての編集操作は Undo で元に戻せる |
| 操作方法 | ドラッグ＆ドロップ・右クリックメニュー・キーボードショートカットの 3 系統 |
| 状態表示 | ステータスバーに現在ページ番号・総ページ数・ズーム率・未保存マークを表示 |
| 最近のファイル | **直近 10 件** のファイル履歴を「ファイル」メニューに表示。**機密漏えい回避のためフルパスは `sessionStorage` に保存**（アプリ／ブラウザ終了で消去）。旧版 `localStorage` 履歴は起動時に自動削除する |

## 5. データモデル

### 5.1 内部データ構造

```
PecoDocument
├── filePath: string                    // 元 PDF のファイルパス
├── fileName: string                    // ファイル名（表示用）
├── totalPages: number                  // 総ページ数
├── metadata: PDFMetadata               // PDF メタデータ
├── pages: Map<pageIndex, PageData>     // ページデータ（遅延読み込み）
└── mtime?: number                      // ファイルの最終更新時刻（ms）

PageData
├── pageIndex: number                   // ページ番号（0-based）
├── width: number                       // ページ幅（pt）
├── height: number                      // ページ高さ（pt）
├── textBlocks: TextBlock[]             // バウンディングボックス一覧（読み順）
├── isDirty: boolean                    // 未保存の編集があるか
└── thumbnail: string | null            // サムネイル画像（Base64 または Blob URL）

TextBlock
├── id: string                          // 一意識別子（UUID）
├── text: string                        // テキスト内容
├── originalText: string                // 元のテキスト（差分検出用）
├── bbox: BoundingBox                   // バウンディングボックス座標
├── writingMode: "horizontal" | "vertical"  // 文字方向
├── order: number                       // 読み順（0-based）
├── isNew: boolean                      // 新規追加された BB か
├── isDirty: boolean                    // このブロックに未保存の変更があるか
└── children?: string[]                 // グループ化元の BB ID 一覧（グループ化後のみ）

BoundingBox
├── x: number                           // 左上 X 座標（pt）
├── y: number                           // 左上 Y 座標（pt）
├── width: number                       // 幅（pt）
└── height: number                      // 高さ（pt）

UndoStack（ストアで管理）
├── undoStack: Action[]                 // Undo 可能な操作（max 100）
├── redoStack: Action[]                 // Redo 可能な操作
└── Action: { type: 'update_page', pageIndex, before: PageData, after: PageData }
```

### 5.2 データライフサイクル

```
         読み込み                 編集                    保存
PDF ──────────→ PecoDocument ──────────→ 編集済み状態 ──────────→ PDF
 (ファイル)       (メモリ)        (メモリ上で変更)        (ファイル)

  ・PDF パーサーが              ・TextBlock の text,     ・PDF ライターが
    テキストレイヤーを解析        bbox, order を変更       テキストレイヤーを
  ・TextBlock[] に変換          ・Undo スタックに記録      再構築して書き出し
```

## 7. 技術選定（確定）

| レイヤー | 選定技術 | 選定理由 |
|---------|---------|---------|
| フレームワーク | **Tauri v2** | Rust ベースの軽量・高速な実行環境。v2 移行により、プラグインシステムの柔軟性とセキュリティが向上。 |
| フロントエンド | **React 19** + **TypeScript** | 最新の React 19 を採用。Concurrent Rendering により大規模ドキュメントでも快適な操作感を実現。 |
| 状態管理 | **Zustand 5** | 超軽量な状態管理。Undo/Redo パックとの統合が容易で、ボイラープレートを最小化。 |
| PDF 解析・表示 | **pdfjs-dist 5.5** | 最新の pdf.js を採用。Vite 7 環境下での WebWorker 動作を Blob URL 戦略で最適化。 |
| PDF 編集・保存 | **@cantoo/pdf-lib 2.6** | pdf-lib のメンテナンス継続フォーク。CMap/CID フォント周りのバグ修正と Node/ブラウザ両対応を享受しつつ、純 JS で軽量な PDF 操作と Content Stream 再構築による正確な読み順（Content Stream 順）制御を実現。 |
| ビルドツール | **Vite 7** | 次世代の高速ビルド。WebWorker の `?url` インポートによる Worker 分離戦略を採用。 |

### 技術的な重要事項（確定済み）

- **T-01: 高速レンダリング戦略**: WebWorker を Blob URL 化して確実に別スレッドで実行。パース処理による UI スレッドのブロックを皆無にする。
- **T-02: メモリキャッシュ**: `PDFPageProxy` をメモリ上にキャッシュ（`pageProxyCache`）し、ページ切り替えを 0ms に短縮。
- **T-03: Shared Proxy**: 同一ファイルへの複数リクエストを `Promise` ベースのキャッシュで一本化。無駄な PDF 解析コストを排除。
- **T-04: Asset Protocol**: Tauri v2 の `asset://` プロトコルを適切に構成（CSP 設定含む）。Range Request による大容量 PDF の部分読み込みを完全サポート。
- **T-05: Path Normalization**: Windows 環境特有のパス区切り（`\`）を正規化し、Tauri バックエンド/アセットプロトコルとの整合性を確保。

## 8. リスク・課題

| # | リスク | 影響度 | 発生可能性 | 対策案 |
|---|--------|--------|-----------|--------|
| R-01 | 数千ページ PDF でのメモリ逼迫 | 高 | 中 | 遅延読み込み + LRU キャッシュ（50 ページ）+ Worker スレッド。早期にプロファイリングで検証 |
| R-02 | 縦書き OCR の BB 座標解析が不正確 | 中 | 高 | pdf.js の縦書き対応状況を PoC で検証。不足なら独自パーサーを検討 |
| R-03 | PDF テキストレイヤーの形式が多様 | 高 | 高 | フォント埋め込み・CID フォント・Type3 フォント等を事前にサンプル検証。対応範囲を明確に定義 |
| R-04 | グループ化後の BB 座標計算で文字がズレる | 中 | 中 | 外接矩形の自動計算 + 手動微調整機能 |
| R-05 | 保存時にオリジナル PDF の画質劣化 | 高 | 中 | テキストレイヤーのみ差し替え。画像ストリームは無変換。保存前後のバイナリ検証をテストに含める |
| R-06 | pdf-lib で読み順制御ができない | 高 | 中 | Content Stream 内のテキストオブジェクト順序を制御する方法を PoC で検証。不可なら mupdf に切替 |
| R-07 | 大量ページの保存処理に時間がかかる | 中 | 高 | 未変更ページはストリームコピーで高速化。Worker で非同期処理。プログレスバーで体感時間を軽減 |

## 9. 用語定義

| 用語 | 定義 |
|------|------|
| OCR テキスト | PDF の透明テキストレイヤーに格納された文字情報。PDF ビューアでのコピペ・検索に使用される |
| テキストレイヤー | PDF の画像の上に重ねられた透明なテキスト層。視覚的には見えないが、選択・コピー・検索が可能 |
| バウンディングボックス（BB） | OCR テキストの位置・範囲を示す矩形領域。1 文字〜複数行にわたる場合がある |
| グループ化 | 複数のバウンディングボックスを 1 つに統合する操作。テキストは結合され、座標は外接矩形になる |
| 読み順 | PDF の Content Stream 内でテキストオブジェクトが出現する順序。コピペ時のテキスト出力順を決定する |
| 遅延読み込み（Lazy Loading） | 必要なページだけをオンデマンドでメモリに展開する手法 |
| LRU キャッシュ | Least Recently Used。最も長く使われていないデータから順に破棄するキャッシュ戦略 |
| 仮想スクロール | DOM に全要素を描画せず、可視範囲の要素のみを動的に描画する手法 |
| 外接矩形 | 複数の矩形を全て包含する最小の矩形 |
| Content Stream | PDF 内部でページの描画命令を記述するストリーム。テキストの出現順が読み順を決定する |
