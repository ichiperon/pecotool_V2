# OCR 機能実装計画書（Windows OCR 優先版）

## 確定事項一覧

| 項目 | 決定内容 |
|---|---|
| OCRエンジン | Windows Media OCR (Windows 10/11 標準機能) ※将来的に PaddleOCR への切り替えも考慮 |
| 動作方式 | Rust バックエンド経由で Windows ランタイム API を呼び出し |
| 実行スコープ | 現在ページ単体 / 全ページ一括（両対応） |
| 縦書き対応 | **必須**（Windows OCR の結果解析 + アスペクト比 + 言語特性による判定） |
| 1000ページ対応 | **必須**（メモリ保護のため、未保存データも IndexedDB へ一時退避し LRU を有効化する） |
| 全ページOCRのUndo | **なし**（実行前に確認ダイアログ表示） |
| キャンセル機能 | あり（全ページ実行中のみ） |
| OCRゼロPDF自動検出 | PDFロード後に検出、全ページOCRを自動提案 |

## アーキテクチャ（Windows OCR 版）

```
[Frontend（React）]
    │ ① ページを canvas に描画 → PNG を一時ディレクトリに保存
    │ ② invoke("run_ocr", { image_path, ... })
    ▼
[Rust（lib.rs）]
    │ ③ Windows.Media.Ocr API を使用して文字認識を実行
    │ ④ 結果を JSON 文字列としてフロントエンドに返却
    ▼
[Frontend]
    ⑤ TextBlock に変換 → ストアに反映
    ⑥ **メモリ管理**: ページ数が多い場合、古い編集済みデータを IndexedDB に一時保存し、メモリから解放
    ⑦ 一時 PNG ファイルを削除
```

## 縦書き判定の改善ロジック

現状の `h > w * 1.5` は 1 文字の際に誤判定するため、以下の複合判定を導入する。
1.  **単語アスペクト比**: `height > width * 1.2` かつ `height > フォントサイズ閾値`。
2.  **文字間隔・並び**: 同一ライン（X座標が近い）に連続する文字がある場合、縦書きと判定。
3.  **Windows OCR 内部情報**: `OcrLine` の `TextAngle` やワードの配置順序を利用。

## 1000ページ対応（メモリ管理戦略）

現在の `pecoStore` は `isDirty: true` のページをメモリに保持し続けるため、1000ページ OCR を行うとクラッシュする。これを回避するため以下のフローを導入する。

1.  **段階的永続化**: 全ページ OCR 中、一定ページ数（例: 50ページ）ごとに、編集内容を IndexedDB の `temporary_changes` ストアに保存する。
2.  **LRU の強制適用**: IndexedDB に保存済みのデータであれば、`isDirty` であってもメモリ（Map）から削除可能とする。
3.  **保存時の統合**: 最終的な PDF 保存時に、「IndexedDB 内の変更点」と「メモリ内の変更点」を統合して PDF に書き込む。

## 実装ステップ

### Step 1: 基礎修正（即時実施）
- `useOcrEngine.ts` の一時ファイルパス構築ミス（Windowsセパレータ問題）の修正。
- Rust 側での縦書き判定精度の向上（1文字判定の改善）。

### Step 2: メモリ管理の実装
- `pecoStore.ts` に「Dirtyデータの外部退避（IndexedDB）」メカニズムを追加。
- 1000ページ処理してもメモリ使用量が一定（例: 200MB以下）に保たれることを確認。

### Step 3: OCR エンジン強化
- Windows OCR の精度が不足する場合、元の計画通り PaddleOCR サイドカーを導入し、バックエンドの `run_ocr` を差し替える。
